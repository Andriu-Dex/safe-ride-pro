\section{Introducci\'on}

Este documento presenta la Especificaci\'on de Requisitos de Software (ERS) del sistema \textbf{SafeRidePro}, una plataforma web orientada a coordinar viajes compartidos seguros entre estudiantes de una misma instituci\'on educativa. El sistema incorpora verificaci\'on institucional, publicaci\'on y solicitud de viajes, seguimiento operativo, mecanismos de confianza comunitaria y capacidades de auditor\'ia administrativa.

La estructura del documento sigue la organizaci\'on sugerida por el est\'andar \textit{IEEE Std 830-1998} para especificaciones de requisitos, adaptada al alcance acad\'emico del proyecto y al estado real del producto. El documento se concentra en dejar claramente establecidos los requisitos funcionales y no funcionales del sistema, de modo que puedan servir como fuente formal para el an\'alisis de requisitos, el dise\~no de casos de prueba y la verificaci\'on posterior del software.

\subsection{Prop\'osito}

Definir de manera clara, completa y verificable los requisitos funcionales y no funcionales de SafeRidePro, proporcionando una base documental confiable para:

\begin{itemize}[leftmargin=1.2cm]
    \item el desarrollo y mantenimiento del sistema,
    \item el dise\~no de casos de prueba funcionales, de integraci\'on y de sistema,
    \item la trazabilidad entre necesidades del producto y actividades de validaci\'on,
    \item la evaluaci\'on acad\'emica y la preparaci\'on de entregables posteriores.
\end{itemize}

\subsection{Alcance}

La presente especificaci\'on cubre el producto web SafeRidePro en su fase funcional actual. Incluye requisitos relacionados con:

\begin{itemize}[leftmargin=1.2cm]
    \item acceso e identidad institucional,
    \item perfil e incorporaci\'on institucional,
    \item onboarding de conductor y revisi\'on administrativa,
    \item gesti\'on de veh\'iculos,
    \item creaci\'on, b\'usqueda, solicitud y operaci\'on de viajes,
    \item tracking operativo v1,
    \item ejecuci\'on operativa del viaje por pasajero,
    \item confianza comunitaria, reportes, sanciones y apelaciones,
    \item auditor\'ia administrativa.
\end{itemize}

Quedan fuera del alcance de esta entrega:

\begin{itemize}[leftmargin=1.2cm]
    \item pagos integrados,
    \item aplicaci\'on m\'ovil nativa,
    \item tracking GPS fino tipo ridesharing comercial,
    \item notificaciones avanzadas,
    \item pruebas de carga y seguridad avanzada como entregable principal.
\end{itemize}

\section{Descripci\'on general}

\subsection{Perspectiva del producto}

SafeRidePro es una aplicaci\'on web modular que permite coordinar viajes compartidos entre estudiantes verificados de una misma instituci\'on. El sistema no es un servicio de transporte comercial tradicional, sino una plataforma restringida a usuarios con dominio institucional autorizado.

La plataforma se apoya en una arquitectura web compuesta por frontend, API backend y base de datos relacional. Adem\'as, incorpora un modelo multiinstitucional con identidad global del usuario y contexto operativo por membres\'ia activa.

\subsection{Caracter\'isticas de los usuarios}

Los tipos principales de usuario del sistema son:

\begin{itemize}[leftmargin=1.2cm]
    \item \textbf{Pasajero:} estudiante verificado que busca viajes, solicita cupos, participa en trayectos y emite calificaciones o reportes.
    \item \textbf{Conductor:} estudiante verificado y aprobado administrativamente para publicar y operar viajes.
    \item \textbf{Administrador institucional:} usuario con privilegios para revisar conductores, reportes, sanciones y apelaciones dentro de su instituci\'on.
    \item \textbf{Superadministrador:} actor global encargado de administraci\'on superior, cat\'alogos e instituciones.
\end{itemize}

\subsection{Restricciones}

Las principales restricciones del sistema son:

\begin{itemize}[leftmargin=1.2cm]
    \item los usuarios deben pertenecer a dominios institucionales autorizados,
    \item los viajes solo son visibles dentro de la misma instituci\'on en esta fase,
    \item los pagos no forman parte del alcance funcional actual,
    \item el tracking en tiempo real avanzado no est\'a implementado en esta fase,
    \item la operaci\'on del viaje debe respetar la coherencia entre estado global del viaje, estado de solicitud y estado operativo del pasajero.
\end{itemize}

\subsection{Suposiciones y dependencias}

Se asume que:

\begin{itemize}[leftmargin=1.2cm]
    \item el usuario dispone de acceso web y conexi\'on a internet,
    \item el dominio institucional del correo est\'a previamente registrado en el sistema,
    \item los servicios de correo y almacenamiento de evidencias se encuentran operativos,
    \item el entorno de ejecuci\'on backend, web y base de datos est\'a disponible para la operaci\'on del sistema,
    \item la gesti\'on administrativa de conductores y reportes se realiza por usuarios autorizados.
\end{itemize}

\section{Requisitos espec\'ificos}

\subsection{Requisitos comunes de interfaces}

\subsubsection{Interfaces de usuario}

El sistema debe proporcionar una interfaz web moderna, clara, responsive y consistente, con navegaci\'on por m\'odulos y jerarqu\'ia visual adecuada. Las pantallas principales incluyen autenticaci\'on, perfil, conductor, veh\'iculos, viajes, confianza y auditor\'ia.

\subsubsection{Interfaces de hardware}

No aplica hardware especializado para esta fase. El sistema requiere un equipo cliente con navegador web moderno y conectividad a internet.

\subsubsection{Interfaces de software}

El producto interact\'ua principalmente con:

\begin{itemize}[leftmargin=1.2cm]
    \item frontend web en Next.js,
    \item API backend en NestJS,
    \item base de datos PostgreSQL,
    \item ORM Prisma,
    \item proveedor SMTP para verificaci\'on y recuperaci\'on de cuenta,
    \item servicios geogr\'aficos basados en Leaflet y Geoapify.
\end{itemize}

\subsubsection{Interfaces de comunicaci\'on}

La comunicaci\'on entre cliente y servidor se realiza mediante servicios HTTP REST autenticados. El sistema tambi\'en emite actualizaciones operativas en tiempo real para el seguimiento del viaje en su versi\'on actual.
