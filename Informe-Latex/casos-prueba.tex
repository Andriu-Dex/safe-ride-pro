\section{Introducci\'on}

Este documento presenta el dise\~no de los casos de prueba para \textbf{SafeRidePro}, tomando como base los requisitos funcionales definidos previamente en el informe de especificaci\'on de requisitos. La finalidad del entregable es dejar establecida una bater\'ia inicial de verificaci\'on que permita validar el comportamiento funcional del sistema desde una perspectiva de integraci\'on entre m\'odulos y de comportamiento global de la plataforma.

La propuesta se concentra en los requisitos funcionales RF-001 a RF-024. Para cada uno de ellos se dise\~nan al menos tres casos de prueba, incluyendo escenarios positivos y negativos. Los casos contemplan la interacci\'on entre interfaz, servicios, reglas de negocio, persistencia y restricciones operativas reales del producto.

\subsection{Prop\'osito}

Definir un conjunto trazable de casos de prueba que permita:

\begin{itemize}[leftmargin=1.2cm]
    \item Validar cada requisito funcional especificado en la ERS.
    \item Identificar escenarios de aceptaci\'on y rechazo.
    \item Preparar la futura ejecuci\'on manual o semiautomatizada de pruebas.
    \item Servir como base para matrices de trazabilidad, registro de incidencias y evidencias.
\end{itemize}

\subsection{Alcance}

El alcance del documento incluye:

\begin{itemize}[leftmargin=1.2cm]
    \item Acceso institucional, verificaci\'on, autenticaci\'on y recuperaci\'on.
    \item Perfil, contexto institucional, conductor y veh\'iculos.
    \item Viajes, solicitudes, ejecuci\'on operativa y tracking v1.
    \item Confianza comunitaria, reportes, sanciones, apelaciones y auditor\'ia.
\end{itemize}

No forma parte de este entregable la ejecuci\'on real de las pruebas, la automatizaci\'on formal de scripts, la medici\'on de rendimiento o la elaboraci\'on de reportes de defectos.

\section{Enfoque de dise\~no}

\subsection{Criterios utilizados}

Los casos de prueba se dise\~naron con los siguientes criterios:

\begin{itemize}[leftmargin=1.2cm]
    \item Cobertura directa de cada requisito funcional identificado en la ERS.
    \item Equilibrio entre escenarios positivos y negativos.
    \item \'Enfasis en reglas de negocio, permisos, persistencia de estados y consistencia entre pantallas y backend.
    \item Orientaci\'on a pruebas de integraci\'on y pruebas del sistema, evitando casos de unidad.
\end{itemize}

\subsection{Tipos de prueba considerados}

\subsection{Convenci\'on de identificadores}

\begin{itemize}[leftmargin=1.2cm]
    \item Los requisitos funcionales conservan el identificador del informe base: RF-001 a RF-024.
    \item Cada caso de prueba se identifica como CP-XXX.
    \item La relaci\'on requisito-caso se mantiene en orden secuencial para facilitar la trazabilidad.
\end{itemize}
