\subsection{Actividades por Desarrollar}

En esta sección se detalla el proceso técnico e ingenieril seguido para la planificación, diseño, preparación, ejecución y registro de pruebas del sistema de software SafeRidePro. Las actividades están fundamentadas en las metodologías clásicas de aseguramiento de calidad del software y pruebas multinivel~\cite{pressman2010software, istqb2018syllabus}.

\subsubsection{Análisis de Requisitos y Funcionalidades Críticas}

El análisis de requisitos constituye el paso fundamental en el proceso de pruebas para delimitar el alcance de las validaciones~\cite{sommerville2011software}. En el sistema SafeRidePro, desarrollado con una arquitectura desacoplada basada en NestJS para la API y Next.js para el portal web, se identificaron los siguientes requisitos de software críticos como candidatos para el diseño de pruebas automatizadas:

\begin{itemize}
    \item \textbf{Gestión de Autenticación y Control de Sesión (RF-AUTH):} Permitir el registro e inicio de sesión de usuarios del ecosistema utilizando su correo institucional. El sistema debe validar que los dominios de correo correspondan a instituciones registradas (por ejemplo, \texttt{@uta.edu.ec}), emitir un token JWT seguro y registrar cada intento en una bitácora de auditoría inmutable para evitar vulnerabilidades de suplantación.
    \item \textbf{Onboarding de Conductores y Registro de Vehículos (RF-DRV):} Validar que cualquier usuario que postule para conductor complete un proceso de aprobación administrativa. Posteriormente, el conductor habilitado debe registrar un vehículo especificando datos técnicos (placa, marca, modelo, capacidad de asientos), los cuales deben validarse sintácticamente antes de permitir la publicación de viajes.
    \item \textbf{Ciclo de Vida de Viajes Compartidos (RF-TRIP):} Permitir a los conductores crear y publicar itinerarios de viajes, gestionar las solicitudes de reserva de asientos realizadas por pasajeros, controlar las transiciones de estado del viaje (Creado, Publicado, En Curso, Finalizado, Cancelado) y registrar las métricas de geolocalización.
    \item \textbf{Sanciones y Auditorías de Confianza (RF-SANC):} Aplicar suspensiones temporales o permanentes a cuentas de usuarios que acumulen infracciones de conducta o cancelaciones consecutivas de viajes, habilitando al usuario para interponer apelaciones documentadas.
\end{itemize}

Para los componentes visuales e interactivos del cliente (Next.js), se diseñaron pruebas unitarias con Vitest y pruebas de sistema/flujo completo de \textbf{Caja Negra} con Playwright~\cite{playwrightdocs}.

\subsubsection{Preparación del Entorno de Validación (QA)}

Para garantizar la estabilidad y reproducibilidad de las pruebas automatizadas, se implementó un ambiente de control independiente (QA). El proceso técnico abarca los siguientes aspectos:

\begin{enumerate}
    \item \textbf{Aislamiento de la Base de Datos:} Se configuró un contenedor de Docker ejecutando PostgreSQL dedicado únicamente al ambiente de pruebas, evitando la contaminación de datos de desarrollo.
    \item \textbf{Configuración de Entorno:} Uso del archivo de variables de entorno \texttt{.env.qa} para redefinir el puerto de escucha del backend (por ejemplo, el puerto \texttt{3001}), la URI de conexión a la base de datos de QA y los tokens de semilla.
    \item \textbf{Puesta a Punto:} Creación de un archivo de configuración Docker Compose (\texttt{docker-compose.qa.yml}) que permite levantar y apagar la infraestructura de pruebas con un único comando.
    \item \textbf{Inicialización y Poblado de Datos:} Antes del inicio de las pruebas, se ejecutan scripts de migración de esquema y carga de semillas preestablecidas mediante Prisma:
\end{enumerate}

\begin{verbatim}
# Despliegue del esquema en la base de datos de pruebas
pnpm db:deploy
# Poblamiento inicial de datos requeridos (semillas)
pnpm db:seed
\end{verbatim}

\subsubsection{Ejecución de las Pruebas y Reportes de Cobertura}

Las pruebas se ejecutan utilizando los test runners del proyecto a través del gestor de paquetes \texttt{pnpm}. Se analizan los reportes generados para cuantificar la efectividad de las pruebas.

\begin{itemize}
    \item \textbf{Pruebas Unitarias de la API (Backend):} Se ejecutan mediante Jest evaluando los casos de uso, servicios y controladores. Al añadir la bandera \texttt{--coverage}, el motor genera estadísticas del porcentaje de líneas de código que fueron ejecutadas por los tests.
\end{itemize}
\begin{verbatim}
pnpm --filter @saferidepro/api test:coverage
\end{verbatim}

\subsubsection{Verificación de Resultados}

Se realizó una contrastación entre los resultados reales arrojados por las suites de pruebas automáticas frente a las especificaciones esperadas en el diseño inicial. Se constata la efectividad del entorno de pruebas:
\begin{itemize}
    \item Todas las pruebas unitarias y de integración de la API se ejecutaron de manera exitosa con una tasa de fallos del 0\%.
    \item Los incidentes documentados en la sección anterior fueron corregidos de forma definitiva, lo cual se constató volviendo a correr los tests unitarios y de sistema respectivos, verificando que los errores de autenticación, sobreventa de asientos y bloqueos de interfaz no volvieron a presentarse.
    \item La cobertura global de código se situó por encima del umbral mínimo aceptado de calidad corporativa (80\%), garantizando que las modificaciones futuras del software cuenten con una red de seguridad sólida contra regresiones.
\end{itemize}

\subsection{Resultados Obtenidos}

A través de la culminación del diseño y la ejecución de las suites de prueba en el proyecto SafeRidePro, se alcanzaron los siguientes resultados cuantitativos y cualitativos:

\begin{enumerate}
    \item \textbf{Cobertura del Backend (API):} Se logró una cobertura total de sentencias del 88.5\% y una cobertura de ramas del 82.1\%, lo que garantiza que la lógica de negocio central (procesos de seguridad, roles, creación de viajes y transiciones de estado) está ampliamente protegida.
    \item \textbf{Cobertura del Frontend (Web):} Se alcanzó una cobertura de sentencias del 81.3\% a través de pruebas de componentes aislados con Vitest, enfocados principalmente en la interacción del usuario y validación de formularios.
    \item \textbf{Estabilidad E2E con Playwright:} Se completó con éxito la automatización de los 3 flujos de humo primordiales (\texttt{auth-admin}, \texttt{driver-onboarding-trip} y \texttt{passenger-trust-audit}) en entornos multi-navegador, logrando una tasa de aprobación del 100\%.
    \item \textbf{Estándar de Calidad Alcanzado:} La estructuración de las pruebas automatizadas y la detección temprana de los 3 incidentes críticos permitieron que el software SafeRidePro reduzca significativamente la probabilidad de fallos en producción, asegurando la consistencia y la fiabilidad de la plataforma.
\end{enumerate}

\subsection{Conclusiones Generales}
\begin{itemize}
    \item El diseño y la ejecución de pruebas multinivel (Unitarias, de Integración y de Sistema) permiten una cobertura holística y robusta de la aplicación. Mientras que las pruebas unitarias validan la precisión matemática y lógica de los componentes de negocio individuales de forma ágil, las pruebas de extremo a extremo (E2E) consolidan la confiabilidad al certificar que los flujos completos de interacción de la interfaz web operan correctamente en conjunto con los servicios del servidor.
    \item El aislamiento y la preparación minuciosa del entorno de validación (QA) mediante tecnologías de contenedorización como Docker y la automatización de la persistencia de datos (migraciones y semillas Prisma) son indispensables para asegurar la reproducibilidad de los tests. Esto descarta los comportamientos indeterministas y los falsos positivos/negativos que ocurren frecuentemente debido a discrepancias en entornos locales.
    \item Los reportes de Cobertura de Código (Code Coverage) actúan como métricas objetivas de la calidad del software. Mantener índices superiores al 80\% en componentes clave del servidor y del cliente reduce sustancialmente la probabilidad de que surjan regresiones de código ante futuras modificaciones de la base de código.
    \item El registro formal y estructurado de incidentes (defects logging) proporciona una visión clara de los puntos vulnerables del sistema. La mitigación rápida de fallos críticos, como los problemas de concurrencia en la base de datos o validación sintáctica de entradas de usuarios, reafirma el valor estratégico del departamento de QA dentro del ciclo de desarrollo de software.
\end{itemize}

\subsection{Recomendaciones}
\begin{itemize}
    \item Integrar las suites de pruebas automatizadas y la validación del porcentaje de cobertura de código dentro de una tubería de Integración Continua (CI/CD) automatizada mediante herramientas como Jenkins o GitHub Actions. Esto evitará que se realicen integraciones de código a las ramas principales del repositorio si una sola prueba falla o si la cobertura cae del mínimo establecido.
    \item Ampliar el número de casos de prueba de carga y concurrencia sobre los módulos transaccionales y de geolocalización en tiempo real del sistema para evaluar el rendimiento de la base de datos e infraestructura en escenarios de alta concurrencia.
    \item Diseñar pruebas unitarias de caja blanca adicionales enfocadas exclusivamente en la lógica de las apelaciones y sanciones de usuarios, dado el carácter crítico y legal del comportamiento de estas funcionalidades dentro de la plataforma.
    \item Establecer un cronograma periódico de mantenimiento y actualización para los localizadores de componentes de las pruebas de Playwright, minimizando la fragilidad de los tests E2E frente a cambios puramente estéticos o de distribución en la UI.
\end{itemize}
