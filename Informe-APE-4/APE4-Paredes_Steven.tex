\subsubsection{Análisis de Requisitos y Funcionalidades Críticas}

El análisis de requisitos constituye el paso fundamental en el proceso de pruebas para delimitar el alcance de las validaciones~\cite{sommerville2011software}. En el sistema SafeRidePro, desarrollado con una arquitectura desacoplada basada en NestJS para la API y Next.js para el portal web, se identificaron los siguientes requisitos de software críticos como candidatos para el diseño de pruebas automatizadas:

\begin{itemize}
    \item \textbf{Gestión de Autenticación y Control de Sesión (RF-AUTH):} Permitir el registro e inicio de sesión de usuarios del ecosistema utilizando su correo institucional. El sistema debe validar que los dominios de correo correspondan a instituciones registradas (por ejemplo, \texttt{@uta.edu.ec}), emitir un token JWT seguro y registrar cada intento en una bitácora de auditoría inmutable para evitar vulnerabilidades de suplantación.
    \item \textbf{Onboarding de Conductores y Registro de Vehículos (RF-DRV):} Validar que cualquier usuario que postule para conductor complete un proceso de aprobación administrativa. Posteriormente, el conductor habilitado debe registrar un vehículo especificando datos técnicos (placa, marca, modelo, capacidad de asientos), los cuales deben validarse sintácticamente antes de permitir la publicación de viajes.
    \item \textbf{Ciclo de Vida de Viajes Compartidos (RF-TRIP):} Permitir a los conductores crear y publicar itinerarios de viajes, gestionar las solicitudes de reserva de asientos realizadas por pasajeros, controlar las transiciones de estado del viaje (Creado, Publicado, En Curso, Finalizado, Cancelado) y registrar las métricas de geolocalización.
    \item \textbf{Sanciones y Auditorías de Confianza (RF-SANC):} Aplicar suspensiones temporales o permanentes a cuentas de usuarios que acumulen infracciones de conducta o cancelaciones consecutivas de viajes, habilitando al usuario para interponer apelaciones documentadas.
\end{itemize}

\subsubsection{Marco Teórico y Fundamentos Metodológicos de Pruebas}

El aseguramiento de la calidad del software (Software Quality Assurance - SQA) y el proceso de pruebas (Testing) constituyen disciplinas críticas dentro del ciclo de vida del desarrollo de software (SDLC)~\cite{pressman2010software}. Conforme lo establece la norma ISO/IEC 25010, la calidad del producto de software se evalúa bajo atributos clave como adecuación funcional, fiabilidad, seguridad y usabilidad~\cite{iso25010}. Las pruebas de software se definen como la ejecución controlada del sistema para identificar discrepancias entre los resultados esperados y el comportamiento real~\cite{istqb2018syllabus}.

\subsubsection*{Niveles de Pruebas y la Pirámide de Automatización}
Para estructurar un proceso de control de calidad exhaustivo y de costo óptimo, se adopta el modelo de la pirámide de automatización de pruebas:
\begin{itemize}
    \item \textbf{Pruebas Unitarias (Unit Testing):} Representan la base de la pirámide. Evalúan componentes o unidades lógicas individuales (funciones, clases, métodos) de forma aislada, utilizando técnicas de simulación (mocking) para desacoplar dependencias externas (bases de datos, servicios externos). Son de ejecución rápida y bajo costo computacional~\cite{jestdocs}.
    \item \textbf{Pruebas de Integración (Integration Testing):} Comprueban la correcta interacción y comunicación entre diferentes módulos de la aplicación, como la comunicación entre el caso de uso del servidor y la base de datos a través del ORM Prisma~\cite{prismadocs}.
    \item \textbf{Pruebas de Sistema / End-to-End (E2E):} Corona de la pirámide. Validan el flujo completo del sistema simulando el comportamiento real de un usuario final interactuando con el navegador web (Chromium/Firefox) contra el servidor backend activo, permitiendo verificar integraciones, interfaces de usuario y concurrencia bajo condiciones reales~\cite{playwrightdocs}.
\end{itemize}

\subsubsection*{Metodologías de Pruebas: Caja Blanca y Caja Negra}
Las técnicas de diseño de pruebas se clasifican según el nivel de visibilidad del código fuente:
\begin{itemize}
    \item \textbf{Pruebas de Caja Blanca (Structural Testing):} Se basan en el análisis de la estructura interna del código. Tienen como meta asegurar coberturas lógicas específicas como:
    \begin{itemize}
        \item \textit{Cobertura de Sentencias (Statement Coverage):} Porcentaje de instrucciones ejecutadas.
        \item \textit{Cobertura de Ramas (Branch Coverage):} Garantía de que cada punto de decisión (estructuras \texttt{if}, \texttt{switch}) tome los valores de verdadero y falso al menos una vez.
        \item \textit{Cobertura de Caminos (Path Coverage):} Validación de todas las rutas posibles de ejecución.
    \end{itemize}
    \item \textbf{Pruebas de Caja Negra (Behavioral Testing):} Diseñadas sin conocimiento de la arquitectura interna del código, enfocadas en las entradas y salidas funcionales. Se emplean técnicas como:
    \begin{itemize}
        \item \textit{Partición de Equivalencias:} División de entradas en clases válidas e inválidas donde el comportamiento del sistema sea idéntico.
        \item \textit{Análisis de Valores Límite:} Selección de datos de prueba en los extremos de las particiones, donde la probabilidad de ocurrencia de errores es más elevada.
        \item \textit{Transición de Estados:} Validación de los cambios de estado en modelos lógicos condicionados por eventos (por ejemplo, el ciclo de vida de un viaje en SafeRidePro).
    \end{itemize}
\end{itemize}

\subsubsection*{Herramientas Tecnológicas Empleadas}
En el ecosistema SafeRidePro se seleccionaron herramientas líderes en la industria para automatizar cada nivel del proceso de pruebas:
\begin{itemize}
    \item \textbf{Jest:} Marco de pruebas en Javascript/TypeScript diseñado para la simplicidad y la velocidad de ejecución. Se utiliza en el backend NestJS debido a su soporte nativo para aserciones avanzadas y mocking de inyección de dependencias~\cite{jestdocs}.
    \item \textbf{Vitest:} Entorno de pruebas unitarias ultrarrápido nativo de Vite. Empleado en el portal Next.js para renderizar e inspeccionar componentes React en un DOM simulado en memoria sin necesidad de levantar navegadores físicos~\cite{vitestdocs}.
    \item \textbf{Playwright:} Herramienta de automatización del navegador que permite simular la experiencia real del usuario sobre navegadores modernos con soporte para aislamiento de contextos, trazas de depuración y reintentos automatizados~\cite{playwrightdocs}.
    \item \textbf{Docker y Docker Compose:} Utilizados para instanciar una base de datos PostgreSQL de calidad de QA de forma predecible y desechable, garantizando la repetibilidad de los entornos de prueba~\cite{dockerdocs}.
\end{itemize}

\subsubsection{Estrategia de Cobertura y Matriz de Trazabilidad}

Para certificar la integridad del sistema SafeRidePro, se estableció una estrategia de verificación bidimensional que asegura cobertura teórica y ejecución automatizada práctica:
\begin{enumerate}
    \item \textbf{Diseño de Cobertura Funcional Exhaustiva (72 Casos):} Se diseñó un conjunto completo de 72 casos de prueba (tres por cada uno de los 24 requisitos funcionales del estándar IEEE 830). Esto asegura que todas las condiciones lógicas de aceptación y rechazo estén cubiertas conceptualmente, incluyendo clases de equivalencia y valores límite.
    \item \textbf{Suite de Ejecución y Automatización (7 Casos Técnicos):} Para la validación física en el entorno de QA, se implementó una suite automatizada compuesta por 7 casos de prueba representativos de alta complejidad técnica. Estos casos ejecutan aserciones dinámicas en el backend (Jest), frontend (Vitest) y flujos de interfaz extremo a extremo (Playwright).

\subsubsection{Preparación del Entorno de Validación (QA)}

Para garantizar la estabilidad y reproducibilidad de las pruebas automatizadas, se implementó un ambiente de control independiente (QA) que replica con gran precisión la infraestructura del sistema real~\cite{dockerdocs}. El proceso técnico e ingenieril abarca los siguientes aspectos fundamentales:

\subsubsection*{Aislamiento y Topología de la Infraestructura de Pruebas}
Se configuró un contenedor Docker dedicado que aloja una instancia limpia del motor relacional PostgreSQL (denominado \texttt{saferidepro-postgres-qa}). Para evitar conflictos de puertos con la instancia local de desarrollo, se enlazó el contenedor de pruebas al puerto externo \texttt{5433} (en contraste al puerto de desarrollo por defecto \texttt{5432}).

El archivo de configuración \texttt{docker-compose.qa.yml} organiza el entorno levantando dos servicios vitales:
\begin{enumerate}
    \item \textbf{postgres-qa:} Motor PostgreSQL que monta un volumen efímero en memoria (\texttt{tmpfs}) para acelerar los ciclos de escritura y borrado recurrentes propios de las suites de prueba.
    \item \textbf{redis-qa:} Servidor en memoria para almacenar las sesiones de usuario temporales, listas negras de tokens caídos y bloqueos de tasa (Rate Limiting).
\end{enumerate}

\subsubsection*{Estrategia de Configuración de Variables de Entorno}
Se diseñó un archivo de configuración exclusivo \texttt{.env.qa} para el entorno de aseguramiento de la calidad, inyectando variables cruciales para el aislamiento lógico de las pruebas:
\begin{itemize}
    \item \texttt{DATABASE\_URL="postgresql://postgres:postgres@localhost:5433/saferidepro\_qa?schema=public"}
    \item \texttt{PORT=3001} (Evita colisiones con el puerto \texttt{3000} que ejecuta la aplicación de desarrollo).
    \item \texttt{JWT\_SECRET="qa\_jwt\_secret\_key\_highly\_secure\_2026"}
    \item \texttt{SEED\_ADMIN\_EMAIL="admin.safety@uta.edu.ec"}
    \item \texttt{SEED\_ADMIN\_PASSWORD="SafetyAdminPassword2026!"}
\end{itemize}

\subsubsection*{Control de Esquema y Estrategia de Poblamiento Semilla (Seeding)}
Antes de ejecutar cualquier suite de pruebas, la tubería de automatización de control limpia completamente el esquema de la base de datos de QA y despliega las tablas mediante el ORM Prisma utilizando la instrucción \texttt{prisma db push} y \texttt{prisma db seed}~\cite{prismadocs}. \subsubsection*{Ficha de Reporte de Defecto 1 (BUG-01)}
\textbf{Identificador:} BUG-01\\
\textbf{Componente / Módulo:} \texttt{LoginUseCase} (API backend, módulo de autenticación).\\
\textbf{Clasificación de Severidad:} Alta (Fallo funcional en módulo de seguridad).\\
\textbf{Prioridad de Resolución:} Alta.\\
\textbf{Descripción del Fallo:} El sistema rechazaba el inicio de sesión legítimo de usuarios cuyas entradas de correo institucional contenían espacios en blanco al inicio o al final del texto. Por ejemplo, al ingresar \texttt{" student@uta.edu.ec "} en la pantalla de login, el backend devolvía una excepción de credenciales inválidas (Código HTTP 401).\\
\textbf{Causa Raíz:} El caso de uso consultaba directamente en la base de datos utilizando la cadena de texto provista por el cliente de forma directa sin realizar un tratamiento de sanitización previo. La base de datos fallaba al realizar la coincidencia exacta de la cadena de caracteres.\\
\textbf{Código de la Solución e Implementación:} Se modificó el servicio de login para aplicar el método de saneamiento de texto sobre el atributo del DTO:
\begin{lstlisting}[language=TypeScript]
// Antes del arreglo:
const user = await this.userRepository.findByEmail(loginDto.email);

// Con la correccion (sanitizacion de entrada):
const sanitizedEmail = loginDto.email.trim().toLowerCase();
const user = await this.userRepository.findByEmail(sanitizedEmail);
\end{lstlisting}
\textbf{Verificación:} Se ejecutó nuevamente la prueba unitaria \texttt{login.use-case.spec.ts} enviando correos con múltiples espacios y tabulaciones, aprobándose de manera exitosa todas las aserciones de tokens generados.

\subsubsection*{Ficha de Reporte de Defecto 2 (BUG-02)}
\textbf{Identificador:} BUG-02\\
\textbf{Componente / Módulo:} \texttt{CreateTripRequestUseCase} (API backend, módulo de reservas).\\
\textbf{Clasificación de Severidad:} Crítica (Fallo transaccional con impacto directo en negocio).\\
\textbf{Prioridad de Resolución:} Crítica.\\
\textbf{Descripción del Fallo:} Cuando dos pasajeros solicitaban la reserva de la última plaza disponible en un viaje compartido de manera concurrente (en la misma fracción de milisegundo), el sistema permitía ambas reservas. Esto resultaba en una sobreventa de asientos del vehículo, excediendo la capacidad física máxima del vehículo declarada por el conductor.\\
\textbf{Causa Raíz:} La lógica de verificación de capacidad disponible y la creación de la reserva se realizaban de forma asíncrona desacoplada. Entre el momento de lectura de asientos libres y la inserción del nuevo registro, no existía un bloqueo de concurrencia a nivel de base de datos, provocando una condición de carrera (Race Condition) bajo transacciones concurrentes por defecto en Postgres (Read Committed).\\
\textbf{Código de la Solución e Implementación:} Se implementó una transacción interactiva con Prisma ORM configurando el nivel de aislamiento de la base de datos a \texttt{Serializable} para evitar la anomalía de lectura de fantasmas o escrituras no repetibles:
\begin{lstlisting}[language=TypeScript]
// Implementacion de transaccion interactiva serializable
await this.prisma.$transaction(async (tx) => {
  const trip = await tx.trip.findUnique({
    where: { id: tripId },
    include: { requests: true }
  });
  const reservedSeats = trip.requests.filter(r => r.status === 'APPROVED').length;
  if (reservedSeats >= trip.vehicle.seats) {
    throw new BadRequestException('El vehiculo no cuenta con asientos disponibles.');
  }
  return tx.tripRequest.create({ data: { tripId, passengerId, status: 'APPROVED' } });
}, {
  isolationLevel: Prisma.TransactionIsolationLevel.Serializable
});
\end{lstlisting}
\textbf{Verificación:} Se diseñó un script de pruebas de estrés concurrentes utilizando Jest ejecutando peticiones HTTP simultáneas por hilos paralelos. Se comprobó que el sistema procesa una sola de manera exitosa y aborta de forma controlada la segunda con un error HTTP 400.

\subsubsection*{Ficha de Reporte de Defecto 3 (BUG-03)}
\textbf{Identificador:} BUG-03\\
\textbf{Componente / Módulo:} Componente \texttt{LeafletMap} (Web UI, visualización cartográfica).\\
\textbf{Clasificación de Severidad:} Media (Bloqueo de automatización del flujo de pruebas).\\
\textbf{Prioridad de Resolución:} Media.\\
\textbf{Descripción del Fallo:} La suite de pruebas de sistema (E2E) con Playwright se bloqueaba con un error de timeout al intentar abrir la vista de seguimiento de viaje. El navegador de pruebas excedía el límite de 30 segundos cargando el mapa geográfico.\\
\textbf{Causa Raíz:} El componente Leaflet interactivo intentaba descargar mosaicos (tiles) de mapas directamente de los servidores de OpenStreetMap de forma externa. Debido a que el ambiente de QA local donde corren las pruebas automáticas está configurado en un entorno aislado sin acceso a la red de internet externa, las peticiones HTTP de los mosaicos fallaban por timeout deteniendo la cola de aserciones de la UI.\\
\textbf{Código de la Solución e Implementación:} Se configuró un mock para el componente de mapa en el entorno de pruebas, inyectando un componente visual estático cuando el navegador expone la variable global \texttt{window.\_\_TESTING\_\_}:
\begin{lstlisting}[language=TypeScript]
// Mock condicional del mapa interactivo
export function TripMap({ coordinates }) {
  if (typeof window !== 'undefined' && window.__TESTING__) {
    return <div data-testid="mock-map">Mapa en Modo de Prueba (QA Aislado)</div>;
  }
  return <LeafletMapContainer center={coordinates} {...props} />;
}
\end{lstlisting}
\textbf{Verificación:} Playwright se re-ejecutó y detectó el \texttt{data-testid="mock-map"} de forma instantánea en menos de 100ms, permitiendo la continuación exitosa del resto de pasos de la suite de pruebas.

Mantener índices superiores al 80\% en componentes clave del servidor y del cliente reduce sustancialmente la probabilidad de que surjan regresiones de código ante futuras modificaciones de la base de código.
    \item El registro formal y estructurado de incidentes (defects logging) proporciona una visión clara de los puntos vulnerables del sistema. La mitigación rápida de fallos críticos, como los problemas de concurrencia en la base de datos o validación sintáctica de entradas de usuarios, reafirma el valor estratégico del departamento de QA dentro del ciclo de desarrollo de software.
\end{itemize}

\subsection{Recomendaciones}
\begin{itemize}
    \item Integrar las suites de pruebas automatizadas y la validación del porcentaje de cobertura de código dentro de una tubería de Integración Continua (CI/CD) automatizada mediante herramientas como Jenkins o GitHub Actions. Esto evitará que se realicen integraciones de código a las ramas principales del repositorio si una sola prueba falla o si la cobertura cae del mínimo establecido.
    \item Ampliar el número de casos de prueba de carga y concurrencia sobre los módulos transaccionales y de geolocalización en tiempo real del sistema para evaluar el rendimiento de la base de datos e infraestructura en escenarios de alta concurrencia.
    \item Diseñar pruebas unitarias de caja blanca adicionales enfocadas exclusivamente en la lógica de las apelaciones y sanciones de usuarios, dado el carácter crítico y legal del comportamiento de estas funcionalidades dentro de la plataforma.
    \item Establecer un cronograma periódico de mantenimiento y actualización para los localizadores de componentes de las pruebas de Playwright, minimizando la fragilidad de los tests E2E frente a cambios puramente estéticos o de distribución en la UI.
\end{itemize}
