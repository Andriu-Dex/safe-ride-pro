\subsection{Instrucciones}
Los estudiantes deberán implantar el sistema desarrollado tanto en un servidor local como en un servidor en la nube, basándose en la investigación realizada sobre ambos enfoques. Deberán seguir los siguientes pasos:

\begin{itemize}
    \item \textbf{Implantación en servidor local}: \\ Seleccionar un servidor de aplicaciones local (por ejemplo, Apache Tomcat, WildFly, GlassFish) y proceder con la instalación, configuración y despliegue del sistema. Asegurarse de que el sistema web y móvil funcione correctamente en este entorno.
    
    \item \textbf{Implantación en servidor en la nube}: \\ Seleccionar una plataforma en la nube (por ejemplo, AWS, Azure, Google Cloud), configurar el entorno en la nube y desplegar el sistema siguiendo las mejores prácticas para la nube. Validar que el sistema esté operativo y accesible desde internet.
    
    \item \textbf{Validación del sistema}:\\ Probar el sistema en ambos entornos (local y nube) para asegurar que funcione correctamente y que los requisitos tanto funcionales como no funcionales se cumplan. Deberán entregar un informe que documente el proceso completo de implantación en ambos entornos, incluyendo capturas de pantalla del sistema en funcionamiento en cada uno de los servidores, los pasos seguidos para la configuración, y cualquier problema que se haya presentado junto con su solución.
\end{itemize}

\subsection{Actividades por Desarrollar}
\subsubsection{Descripción de la arquitectura implantada}
La puesta en marcha de la suite SafeRidePro demandó un examen exhaustivo de la topología distribuida que caracteriza al proyecto. Dado que el software está estructurado como un monorrepo basado en workspaces de pnpm, sus módulos están conceptualmente desacoplados pero vinculados de forma operacional. Esta disposición requiere canales de enlace síncronos e ininterrumpidos para que el backend, la base de datos y la interfaz actúen de forma cohesionada.

La infraestructura del sistema se articula a través de las siguientes capas operativas:
\begin{itemize}[leftmargin=*]
    \item \textbf{Almacenamiento de Datos (Persistencia):} Motor de base de datos relacional PostgreSQL. Para la prueba local se instanció una imagen oficial expuesta dentro de un segmento de red de puente virtual en Docker. En contraste, la versión de nube delegó esta función a una base de datos gestionada en la nube mediante Supabase, resolviendo la persistencia de forma externa.
    \item \textbf{Capa de Negocios (Backend API):} Aplicación construida bajo NestJS encargada de gobernar el flujo de datos. Implementa la autenticación basada en tokens JWT, gestiona la creación de rutas de transporte, ejecuta el procesamiento de transacciones financieras virtuales, audita las actividades de la plataforma y procesa los reportes administrativos.
    \item \textbf{Capa de Presentación (Frontend):} Cliente web dinámico basado en Next.js y React. Se comunica con el backend mediante peticiones asíncronas REST, actuando como interfaz para los diferentes perfiles del sistema: pasajeros, conductores y administradores.
    \item \textbf{Cohesión de Repositorio (Monorrepo):} La organización jerárquica separa los subproyectos en \texttt{apps/api} (backend) y \texttt{apps/web} (frontend), enlazados a través de tipos comunes e interfaces declaradas en el módulo compartido de librerías locales \texttt{packages/shared-types}. Esta interdependencia exige procesos ordenados de compilación previa de dependencias.
    \item \textbf{Orquestación y Enlace:} En el ámbito local se confió en Docker Compose para administrar las redes, volúmenes de disco y puertos expuestos. En la nube, se establecieron servicios PaaS con Render para la ejecución del servidor Node y Supabase para el aprovisionamiento relacional.
\end{itemize}

Esta separación física de las capas asegura que la lógica del negocio permanezca inalterada frente a traslados de entorno, requiriendo únicamente la redefinición de variables operativas de red, tokens y rutas de conexión a bases de datos.

\subsubsection{Preparación del entorno local}
Para estructurar el ambiente local de validación, se configuró una máquina portátil con Windows y el subsistema WSL2 acoplado con Docker Desktop. Este diseño evitó la sobrecarga típica de instalar múltiples servicios y paquetes de base de datos directamente en el sistema operativo anfitrión.

El proceso arrancó con la preparación de un archivo de configuración denominado \texttt{.env.qa} ubicado en la raíz del monorrepo. Este archivo centraliza secretos, claves JWT y variables de red. Un aspecto clave consistió en desviar el puerto de escucha local de la base de datos PostgreSQL dockerizada hacia el puerto 5433, eludiendo colisiones físicas con instancias que pudieran estar ejecutándose nativamente en la máquina.

\evidencia[0.82\linewidth]{ape6_local_01_env_qa.png}{Configuración de parámetros y variables técnicas en el archivo \texttt{.env.qa} para el entorno local.}

La ejecución ordenada para inicializar la suite requirió los siguientes pasos:
\begin{enumerate}[leftmargin=*]
    \item Ubicación y validación sintáctica del archivo \texttt{.env.qa} en la raíz del proyecto.
    \item Ejecución del script de construcción y levantamiento global a través de pnpm:
\begin{verbatim}
corepack pnpm qa:up:build
\end{verbatim}
    \item Comprobación del estado operacional y la asignación de puertos de los contenedores activos:
\begin{verbatim}
corepack pnpm qa:ps
\end{verbatim}
    \item Control de disponibilidad de red interna consultando los siguientes recursos locales:
    \begin{itemize}
        \item \texttt{http://localhost:3000} (Servidor Next.js Frontend)
        \item \texttt{http://localhost:3001/api/health} (Endpoint de salud de API NestJS)
        \item \texttt{http://localhost:3000/healthz} (Endpoint de salud de Next.js Web)
    \end{itemize}
    \item Ingreso mediante navegador a la interfaz de administración pública y validación del inicio de sesión.
\end{enumerate}

La instrucción \texttt{corepack pnpm qa:up:build} automatiza de forma secuencial la descarga de módulos, la compilación de tipados y la inicialización del cliente de Prisma, para luego activar la orquestación en segundo plano por medio del motor Docker Compose.

\subsubsection{Selección y preparación del entorno en nube}
La migración de la arquitectura local de SafeRidePro hacia servidores remotos de acceso público requirió un análisis comparativo y de viabilidad de diversos proveedores cloud gratuitos.

\subsubsection*{1. Evaluación con Oracle Cloud Always Free}
Se diseñó un plan de despliegue fundamentado en una máquina virtual de arquitectura ARM que ejecutaría Ubuntu Server, proyectando replicar exactamente la configuración de Docker Compose expuesta en el entorno local mediante la apertura manual de puertos (3000, 3001, 5433) y la asignación de una IP pública fija. Pese a generar las llaves SSH correspondientes y de configurar las listas de control de seguridad en la consola cloud, la falta de disponibilidad de procesadores Ampere libres en la región del proveedor impidió culminar el aprovisionamiento de la máquina virtual.

\subsubsection*{2. Evaluación con Azure for Students}
De forma complementaria, se exploró el uso de los recursos estudiantiles ofrecidos por Microsoft Azure. Al estudiar las bases de la capa Starter del programa de estudiantes, se observó que el uso concurrente de múltiples servicios web independientes y bases de datos relacionales supera las limitaciones de la cuota mensual gratuita, lo que presentaba el riesgo de cobros indeseados y obligó a descartar el uso de este entorno.

\subsubsection*{3. Selección de la Estrategia Definitiva}
A la luz de estas restricciones de capacidad, se optó por un enfoque arquitectónico basado en Plataformas como Servicio (PaaS) desacopladas. Se seleccionó a Supabase como proveedor de la base de datos PostgreSQL remota y Render para el alojamiento independiente del backend API y el frontend. Este esquema suprimió la sobrecarga de administrar la configuración de firewalls, actualizaciones de kernel o configuraciones de red de una máquina virtual dedicada, enfocando los esfuerzos de desarrollo en estructurar los scripts de compilación de las aplicaciones independientes del monorepo.

\subsubsection{Provisionamiento de la base de datos en Supabase}
El proceso de persistencia en la nube comenzó configurando un clúster relacional administrado en la consola web de Supabase. Una vez concluido el aprovisionamiento del motor de base de datos relacional, se procedió a extraer las credenciales necesarias en la pestaña de configuración.

Debido al uso de Prisma ORM en el backend, la conexión directa presentaba riesgos de saturar las conexiones físicas concurrentes del plan gratuito. Por esta razón, se seleccionó el método de conexión \texttt{Session pooler} en el puerto 5432 (o 6543 para peticiones pool). Esto delega la cola de conexiones físicas a PgBouncer (incorporado en Supabase), optimizando los sockets de conexión y reduciendo drásticamente la latencia originada por transacciones de corta duración.

\subsubsection{Publicación del backend en Render}
La API NestJS del backend de la aplicación se alojó en Render bajo la tipología de servicio \texttt{Web Service}, dándole el nombre identificativo de \texttt{saferidepro-api}. La definición estructural del servicio contempló los siguientes parámetros:
\begin{itemize}[leftmargin=*]
    \item \textbf{Identificador:} \texttt{saferidepro-api}
    \item \textbf{Entorno Técnico:} \texttt{Node}
    \item \textbf{Rama del Repositorio:} \texttt{main}
    \item \textbf{Región de Despliegue:} \texttt{Oregon (US West)}
    \item \textbf{Directorio de Origen (Root Directory):} en blanco (requerido para habilitar el alcance de pnpm a las dependencias transversales en la raíz del monorepo)
    \item \textbf{Verificador de Disponibilidad (Health Check):} \texttt{/api/health}
\end{itemize}

El proceso arrancó enlazando el repositorio de GitHub con la cuenta personal de Render, autorizando el acceso a la lectura del código del proyecto SafeRidePro para implementar compilaciones automáticas basadas en webhooks cada vez que se realice un commit.

\evidencia{ape6_nube_05_render_github_permiso.png}{Configuración de permisos de acceso al repositorio de GitHub desde la plataforma Render.}

\evidencia{ape6_nube_07_render_api_config.png}{Parámetros de despliegue y región configurados para el servicio \texttt{saferidepro-api}.}

El comando de compilación (Build Command) se diseñó para resolver las particularidades del monorepo, instalando pnpm globalmente e invocando la compilación ordenada de los tipos compartidos antes de compilar el backend de NestJS:
\begin{verbatim}
corepack prepare pnpm@latest --activate &&
pnpm install --frozen-lockfile &&
pnpm --filter @saferidepro/shared-types build &&
pnpm --filter @saferidepro/api prisma:generate &&
pnpm --filter @saferidepro/api build
\end{verbatim}

Para el arranque de la API (Start Command), se programó una secuencia lógica que ejecuta las migraciones pendientes de la base de datos de manera automatizada e inyecta los datos semilla de pruebas (seeding) antes de encender el servidor de producción:
\begin{verbatim}
pnpm --filter @saferidepro/api db:deploy &&
pnpm --filter @saferidepro/api db:seed &&
node apps/api/dist/main.js
\end{verbatim}

En el panel de Render se definieron de forma segura las variables de entorno operativas de la API, tales como \texttt{DATABASE\_URL} (apuntando al Session Pooler de Supabase), \texttt{JWT\_SECRET} (para firmar criptográficamente las credenciales) y la variable \texttt{WEB\_APP\_ORIGINS} (con la URL del frontend para habilitar las cabeceras CORS de forma exclusiva).

\subsubsection{Publicación del frontend en Render}
La interfaz web de la aplicación construida en Next.js se desplegó como un servicio web independiente denominado \texttt{saferidepro-web}. La especificación técnica de configuración fue la siguiente:
\begin{itemize}[leftmargin=*]
    \item \textbf{Identificador:} \texttt{saferidepro-web}
    \item \textbf{Entorno Técnico:} \texttt{Node}
    \item \textbf{Rama de Repositorio:} \texttt{main}
    \item \textbf{Región de Despliegue:} \texttt{Oregon (US West)}
    \item \textbf{Directorio de Origen:} vacío
    \item \textbf{Verificador de Disponibilidad (Health Check):} \texttt{/healthz}
\end{itemize}

El comando de compilación (Build Command) estructurado para resolver el enrutamiento de dependencias y construir la aplicación Next.js fue:
\begin{verbatim}
corepack prepare pnpm@latest --activate &&
pnpm install --frozen-lockfile &&
pnpm --filter @saferidepro/shared-types build &&
pnpm --filter @saferidepro/web build
\end{verbatim}

Para asegurar que Next.js escuche peticiones desde internet y se enlace correctamente a la red de contenedores de Render, el comando de inicio (Start Command) se configuró de la siguiente forma:
\begin{verbatim}
node scripts/run-web-next.js start --hostname 0.0.0.0 --port $PORT
\end{verbatim}

Esta configuración inyecta dinámicamente la variable de entorno \texttt{\$PORT} proveída por Render y fuerza la escucha en todas las interfaces (\texttt{0.0.0.0}). Además, se registró la variable pública \texttt{NEXT\_PUBLIC\_API\_BASE\_URL} que apunta a la URL pública del backend desplegado en Render, asegurando que las llamadas asíncronas del cliente web en el navegador del usuario se dirijan a los endpoints remotos correctos.

\subsubsection{Limitaciones observadas}
El despliegue de la suite sobre una infraestructura en la nube bajo los planes de uso gratuitos impuso ciertas restricciones técnicas de rendimiento:
\begin{itemize}[leftmargin=*]
    \item \textbf{Latencia por Suspensión (Cold Start):} Render detiene los contenedores gratuitos si no detectan tráfico HTTP en un periodo de 15 minutos. Al ingresar un usuario tras dicho lapso, la plataforma inicia una fase de arranque en frío que puede demorar entre 45 y 60 segundos antes de resolver la primera petición HTTP \cite{renderfree}.
    \item \textbf{Almacenamiento de Datos Efímero en Render:} La capa gratuita de Render carece de soporte para volúmenes persistentes. Cualquier archivo guardado en el disco local del contenedor (como registros de auditoría locales) se borra definitivamente tras cada redespliegue de código o reinicio periódico del sistema \cite{renderfree}.
    \item \textbf{Límites del Pool en Supabase Free:} El plan gratuito limita de forma estricta las conexiones concurrentes a la base de datos. Sin el uso del Session Pooler, el pool de conexiones nativo de Prisma ORM satura rápidamente los recursos de red de Supabase, provocando errores de conexión intermitentes.
\end{itemize}

Estas limitantes demuestran la diferencia de rendimiento entre el entorno local (donde la base de datos y la API están inmediatamente disponibles) y el entorno cloud gratuito, confirmando que para entornos productivos reales se requiere migrar a planes de recursos dedicados.

\subsection{Resultados Obtenidos}
El proceso de implantación y prueba del sistema SafeRidePro arrojó resultados determinantes para la validación del proyecto en ambos esquemas de infraestructura:

\subsubsection*{1. Paridad Operacional de los Entornos}
Se logró replicar con precisión el funcionamiento de los servicios a través de dos esquemas de infraestructura física distintos. Mientras el entorno local provisto por Docker Compose ofreció una respuesta de red instantánea óptima para las fases rápidas de depuración, el entorno en la nube (Render + Supabase) certificó el correcto comportamiento de la aplicación en condiciones reales de red a través de internet, garantizando que no existan discrepancias lógicas al migrar esquemas de base de datos a través de Prisma.

\subsubsection*{2. Persistencia y Gestión del Pool Relacional}
La provisión de la base de datos PostgreSQL en Supabase, acoplada con el método Session Pooler, posibilitó el control dinámico de los sockets físicos del servidor en la nube. Se constató la ejecución no interactiva del esquema relacional y de las transacciones de inserción inicial (seed) durante la fase de despliegue, logrando mantener el uso de memoria de la base de datos dentro de los umbrales normales asignados por el proveedor.

\subsubsection*{3. Coordinación y Enlace Multiorigen}
Mediante la segmentación física de los servicios en Render, se aisló el cómputo de la interfaz y del backend. La correcta inyección de variables de entorno permitió enlazar el origen del cliente web con el backend NestJS sin generar conflictos por políticas CORS del navegador, logrando que Next.js dirija de forma dinámica sus consultas HTTP a la dirección del servidor API expuesta públicamente.

\subsubsection*{4. Mitigación de Limitaciones de Infraestructura}
Se documentaron las limitaciones de la capa gratuita (demoras de Cold Start y almacenamiento volátil), adaptando la experiencia de pruebas para considerar los tiempos de reinicio de contenedores en Render, logrando un despliegue operativo funcional y verificable sin incurrir en costos económicos.

\subsection{Conclusiones Generales}
\begin{itemize}[leftmargin=*,noitemsep]
    \item La implantación demostró que el acoplamiento de un monorrepo puede gestionarse eficientemente en la nube mediante comandos de construcción filtrados en pnpm, evitando fragmentar el código o crear repositorios independientes para producción.
    \item El uso de Docker Compose en local garantiza la reproducibilidad de la infraestructura en pocos segundos, reduciendo drásticamente la latencia de desarrollo en comparación con las plataformas PaaS gratuitas que sufren de suspensión de red.
    \item La arquitectura basada en Supabase y Render provee una alternativa robusta para el despliegue ágil en fases de staging y pruebas QA, eliminando la sobrecarga operativa y de costos asociada a la provisión de máquinas virtuales y firewalls manuales.
    \item La validación end-to-end realizada certificó que la aplicación SafeRidePro responde correctamente en flujos sensibles de persistencia (Billetera), geolocalización (Seguimiento GPS) y seguridad, consolidando un sistema listo para pruebas de usuario.
\end{itemize}

\subsection{Recomendaciones}
\begin{enumerate}[leftmargin=*,noitemsep]
    \item Utilizar variables de entorno estrictamente delimitadas para desarrollo, QA y producción, evitando que datos de prueba locales contaminen o purguen bases de datos administradas en la nube.
    \item Evaluar el uso de servicios de almacenamiento de objetos externos (como AWS S3 o Cloudinary) para la API de SafeRidePro, con el fin de solventar la pérdida de archivos debida al disco efímero no persistente de Render Free.
    \item Implementar tareas de ping automatizadas (cron jobs) de bajo costo para enviar peticiones periódicas a los endpoints de salud de Render, evitando la suspensión por inactividad y eliminando los retardos por cold start.
    \item En transiciones productivas, migrar la base de datos de Supabase a un esquema de conexión directa bajo una red VPC (Virtual Private Cloud) para optimizar el rendimiento y la seguridad del tráfico.
\end{enumerate}
